# /plan

## Purpose

Run a structured planning session for a new feature or modification. Produces a ready-for-approval plan document. No implementation begins until Commander explicitly confirms.

## Arguments

`[ProjectName] [description]`

- **ProjectName** — must match an entry in `.claude/ProjectEnvironment.md`
- **description** — what you want to build or change (free text, as specific as possible)

## Help

If `[ProjectName]` is `help` (i.e., `/plan help`), print the following and STOP:

```
/plan — Unified planning skill

SYNTAX:
  /plan [ProjectName] [description]

EXAMPLES:
  /plan SyncSpace "Add user authentication with JWT"
  /plan AeroMedica "Refactor triage assessment flow"
  /plan help                            Show this help text

OUTPUT:
  Plan document in:
    03_SubFeatures Implementation/  — for new features
    04_Modification Logs/           — for modifications/refactors
  Status: AWAITING COMMANDER APPROVAL
  No implementation starts until Commander confirms.
```

---

## Steps

1. **Read** `.claude/ProjectEnvironment.md` → locate `PROJECT_ROOT` for [ProjectName]. If not found, halt and ask Commander to add the project first.

2. **Open RoundTable** — append a `## Session [N] — /plan: [description]` entry to today's RoundTable file before proceeding. AM writes the opening line.

3. **Determine destination folder:**
   - New feature or capability → `[PROJECT_ROOT]/Development/[ProjectName]/03_SubFeatures Implementation/`
   - Modification, refactor, or expansion of existing feature → `[PROJECT_ROOT]/Development/[ProjectName]/04_Modification Logs/`
   - If the type is ambiguous from the description, ask Commander before continuing.

4. **Scan for next ORDER number** — list files in the destination folder and determine the next sequential number.

5. **Brainstorm phase — AM leads:**
   Speak in AM's voice. Answer the following:
   - What problem does this solve, and for whom?
   - What are the key constraints (technical, time, scope)?
   - What are the top 3 risks?
   - Are there simpler alternatives that achieve the same goal?
   - What does "done" look like from Commander's perspective?

6. **Design phase — MT leads:**
   Speak in MT's voice (technical only). Answer:
   - Which files, modules, and layers does this touch?
   - What new interfaces, schemas, or data structures are needed?
   - Which existing patterns does this follow (or deliberately break, and why)?
   - Are there any architectural decisions that require MT sign-off?
   - What is the implementation approach in 3–5 concrete steps?

7. **Spec review phase — AS leads:**
   Speak in AS's voice (UX + QA). Answer:
   - What are the acceptance criteria? (List each as a checkbox.)
   - What are the edge cases and failure paths?
   - What does the testing strategy look like?
   - Is there any user-facing change? If yes: UX Smoke Test gate applies.

8. **Workflow phase — AM leads:**
   - Break the implementation into numbered tickets
   - Assign each to a team (Monolith / Syndicate / Arcade)
   - Run a ZCB check: confirm no ticket in this list depends on another team's unfinished output in the same phase
   - List any dependencies between tickets

9. **Write the plan document** to the destination folder:
   - File: `[ORDER]. PLANNED_[name]_[DD-MM-YYYY].md`
   - Use the Plan Document Format below

10. **Halt for Commander approval:**
    Present the plan document summary (problem, solution approach, ticket list).
    State: **"Awaiting Commander approval. No implementation begins until you confirm."**

---

## Plan Document Format

```markdown
# Plan: [Feature/Scope Name]

**Project:** [ProjectName]
**Date:** DD-MM-YYYY
**Status:** AWAITING COMMANDER APPROVAL
**Type:** Sub-Feature | Modification

---

## Problem Statement
[From brainstorm — what problem this solves and for whom, 2–4 sentences]

## Proposed Solution
[MT's design — approach in 3–5 steps, which files are touched]

## Architecture Decisions
[Any MT sign-off items — list each as a decision with rationale. "None required." if not applicable.]

## Acceptance Criteria
- [ ] criterion 1
- [ ] criterion 2

## Edge Cases / Risks
[From AS — failure paths, risks, edge cases to handle]

## Implementation Tickets

| # | Team | Ticket Title | Depends On |
|---|------|--------------|------------|
| 1 | Monolith | [title] | None |
| 2 | Syndicate | [title] | #1 |

## Open Questions
[Anything unresolved that Commander must decide before implementation can begin]

---

*Plan generated by /plan skill — [DD-MM-YYYY]*
*Status: AWAITING COMMANDER APPROVAL*
```

## Output

Plan document path presented to Commander. RoundTable session entry logged. Execution halted pending Commander approval.
