# /document

## Purpose

Generate a structured feature description document from source code. Enforces the Living Documentation Rule by making documentation creation a single, repeatable command rather than a passive reminder.

## Arguments

`[ProjectName] [FeatureName]`

- **ProjectName** — must match an entry in `.claude/ProjectEnvironment.md`
- **FeatureName** — name of the feature or subsystem to document (becomes the folder and file name)

## Help

If `[ProjectName]` is `help` (i.e., `/document help`), print the following and STOP:

```
/document — Feature documentation generator

SYNTAX:
  /document [ProjectName] [FeatureName]

EXAMPLES:
  /document SyncSpace SnapshotSystem
  /document AeroMedica TriageAssessFlow
  /document help                         Show this help text

OUTPUT:
  Development/[ProjectName]/02_FeatureDescription/[FeatureName]/[FeatureName]_Description.md
  Creates if missing. Updates in-place if it exists (living doc).
```

---

## Steps

1. **Read** `.claude/ProjectEnvironment.md` → locate both `PROJECT_ROOT` and `SOURCE_ROOT` for [ProjectName]. If not found, halt and ask Commander to add the project first.

2. **Determine output path:**
   ```
   [PROJECT_ROOT]/Development/[ProjectName]/02_FeatureDescription/[FeatureName]/[FeatureName]_Description.md
   ```
   Check if this file already exists:
   - **Exists:** Read the current file. This is an **update** — preserve accurate existing sections, fill gaps, and correct outdated content. Note `Updated: DD-MM-YYYY` in the footer.
   - **Does not exist:** This is a **creation** — proceed to scan.

3. **L1 Scan** of `[SOURCE_ROOT]` — list directory tree to depth 3. Identify:
   - Files and modules likely related to [FeatureName] (by name, folder path, or inferred context)
   - Entry points, controllers, services, or components matching the feature

4. **L2 Scan** — read the key files identified in L1. For each file:
   - What does this module do?
   - What are its exported functions, classes, or components?
   - What are its inputs and outputs?
   - What does it depend on?

5. **Generate documentation** using only information found during L1/L2 scan.
   - **Do NOT hallucinate:** If a section cannot be populated from the scan, write `[Not found in scan — review manually]`
   - Every technical claim must be traceable to a file read in Steps 3–4

6. **Write the document** to the output path using the Feature Description Format below. Create the folder if it does not exist.

7. **Report** the output file path to Commander.

---

## Feature Description Format

```markdown
# Feature: [FeatureName]

**Project:** [ProjectName]
**Last Updated:** DD-MM-YYYY
**Scan Tier Completed:** L1 | L2
**Source Files Reviewed:**
- `path/to/file1`
- `path/to/file2`

---

## Overview
[2–4 sentences: what this feature does and why it exists. From source code, not assumed.]

## Components

| File / Module | Role | Key Exports |
|---------------|------|-------------|
| `path/to/file` | [what it does] | [exported functions/classes] |

## Trigger Conditions
[What causes this feature to activate — user action, API call, event, scheduled trigger, etc. From source only.]

## Input / Output

| Direction | Type | Description |
|-----------|------|-------------|
| Input | [type] | [what comes in] |
| Output | [type] | [what goes out] |

## Edge Cases
[Found in source — guard clauses, error handling, conditional branches that handle unusual states]
- [edge case 1]
- [edge case 2]

## Related Policies
[Which §-numbered policies govern this feature's behavior, if any. "None" if not applicable.]

## Open Questions
[Anything the scan could not determine — for the team to resolve manually. Leave blank if scan was complete.]

---

*Generated by /document skill*
*Scan tier: L2 — [DD-MM-YYYY]*
```

---

## Output

File path of the created or updated feature description document. If this is an update, the prior version is preserved (overwritten in-place — use git history for rollback if needed).
